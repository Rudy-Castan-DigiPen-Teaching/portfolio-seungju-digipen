Project Pokémon Graph

Interactive p5.js Network

Live Interactive Demo

Try clicking or hovering over the type nodes to check out the matchups. You can filter the attack/defense matchups individually using the mode buttons at the top.

Project Overview

This project is an interactive web app built with p5.jsthat visualizes the 18 Pokemon type matchups. I started it to fix a design problem: classic text tables make it super hard to intuitively grasp complex type relations. I focused on creating a visual solution where users can dynamically filter and explore just the attack and defense matchups they want to see.

Tasks Completed

  • Interactive Network Visualization: I set up 18 type nodes using p5.js and implemented math-based animations so the layout smoothly transitions (lerp) to focus on a selected type via mouse hover and click events.
  • Dynamic Matchup Filtering: I built a state management system that controls the rendered edges in real-time based on the user's choice, adding 'Attack only', 'Defense only', and 'View all' modes.
  • Data-Driven Rendering: I modularized the 18x18 array Type Chart data into objects to calculate damage multipliers (2x, 0.5x, 0x, etc.) between specific types, visualizing them through line colors, thicknesses, and dashed styles.

Reflection

Challenge 1: Visual Clutter from Complex Data
Problem: Rendering every single matchup (edge) between the 18 types on the screen at once caused lines to overlap heavily, ruining readability and causing visual fatigue.
Solution: When a user clicks or hovers over a specific node, that node moves to the center of the screen, and only its related matchup edges are clearly highlighted (Alpha 1.0). I applied a dynamic layout that drops the opacity of the unselected elements (Alpha 0.1 ~ 0.25) so the user can focus purely on the key info.

Challenge 2: Dynamic i18n for Global Accessibility
Problem: Outputting text hardcoded in a single language (Korean) drops accessibility for users from other regions. Plus, I had to consistently translate the text rendered inside both the DOM elements and the p5.js canvas.
Solution: I used the navigator.language API to detect the client's default browser language. Based on this, I designed a multi-language object (i18n) that dynamically maps UI text, canvas node names, and legends, ensuring Korean and English switch naturally depending on the user's environment.
